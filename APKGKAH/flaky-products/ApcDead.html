However, I
can use the Web interface and front panel buttons to power off the load. I
don't think there is any hardware or firmware defect. We run Debian and
Ubuntu, and may not be able to easily switch to RHEL (CentOS may be
possible).
Serial cable If I connect the APC serial cable and run minicom (Linux
counterpart of Hyperterm), I see the login prompt, I can log in, see and
change all configuration parameters. Everything looks normal. If I
configure apcupsd (Ubuntu and Debian, versions 3.14.6-3 and 3.14.8-2
tried), the UPS is monitored normally. On power cut, a shutdown trigger
happens after the configured battery time. However, at the end of the
shutdown sequence, the computer sends a signal to the UPS to turn off the
load (until utility power returns). *This command is being ignored: the
load is never turned off.* I have tried sending the same command from a
battery-powered laptop via the serial port, and the laptop is ignored as
well. No log record is added to show a power down command.
Ethernet, PowerChute PCnet protocol I set the 15-character pass phrase
properly. When killpower is issued, the UPS does register a log entry of
the following form:

UPS: A graceful shutdown process is being used to shut down the load
equipment before the UPS turns off. Initiated by PowerChute ns from
laptop.domain

But after that, *the UPS does not follow through and turn the load off.*

Ethernet, SNMP v1 protocol I set the private domain with write+ access and
registered the IP address allowed to write. *Killpower is again entirely
ignored*, without any log message, even when issued from a battery-powered
laptop.

What we need

- Complete instruction for 100% reliable power down of load after
shutdown of the controlling computer. Either serial cable or SNMP v1/v3 is
ok provided it works reliably.
- *All* (not just the latest) firmwares released between sumx 3.3.1 and
3.7.2 and aos 3.3.4 and 3.7.3, *inclusive*. ftp.apc.com is frequently
down and only the latest is available online.
